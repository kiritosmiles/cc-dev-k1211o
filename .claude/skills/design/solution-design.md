# Solution Design Skill

需求方案设计模板。由 analyst agent 调用，用于将用户需求转化为结构化方案。

---

## 一、适用场景

- 新功能开发方案
- 模块设计
- 跨模块功能设计
- 技术选型建议

---

## 二、分析维度

在生成方案前，必须从以下维度分析需求：

### 1. 需求理解
- 用户真正想解决什么问题？
- 是否与项目目标一致？（参考 project-context.md）
- 是否有隐含需求未明说？

### 2. 架构影响
- 是否影响模块边界？（参考 architecture.md）
- 是否需要新建模块？
- 与现有模块的依赖关系？

### 3. 技术可行性
- 现有技术栈是否支持？（参考 tech-stack.md）
- 需要引入新依赖吗？
- 有无已知限制？（参考 known-issues.md）

### 4. 数据模型
- 涉及哪些实体？
- 实体之间的关系？
- 是否需要新增/修改数据库表？

### 5. API 设计
- 需要哪些接口？
- 与现有 API 是否一致？（参考 api-standard.md）

### 6. UI 设计（如涉及前端）
- 涉及哪些页面/组件？
- 与现有设计系统是否一致？（参考 design-system.md）

### 7. 风险识别
- 对现有功能的破坏风险？
- 性能风险？
- 安全风险？（参考 security-standard.md）

## 三、方案模板

```markdown
## 需求分析

### 用户目标
{一句话}

### 背景
{为什么需要这个功能，当前状态的不足}

### 需求拆解
1. {子需求1}
2. {子需求2}

---

## 技术方案

### 整体思路
{一句话描述技术方案的核心思路}

### 模块设计
| 模块 | 变更类型 | 说明 |
|------|---------|------|

### 数据模型（如有新实体）
```text
EntityName
├── field1: type
├── field2: type
└── relation: → OtherEntity
```

### API 草案（如有新接口）
| Method | Path | Request | Response | 说明 |
|--------|------|---------|----------|------|

### 前端组件（如涉及）
| 组件 | 路径 | 类型 | 状态管理 |
|------|------|------|---------|

---

## 任务拆分

| # | 任务标题 | Agent | 优先级 | 依赖 |
|---|---------|-------|--------|------|

### 依赖图
```text
Task-1 ──→ Task-2 ──→ Task-4
       └─→ Task-3 ──┘
```
(→ = 依赖)

---

## 边界条件
- 空数据: {如何处理}
- 错误状态: {如何处理}
- 并发: {是否需要考虑}
- 权限: {谁可以访问}

---

## 风险清单

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|

---

## 验收条件
- [ ] {条件1}
- [ ] {条件2}

---

## 待确认项（如有）
- [ ] {不确定、需要用户确认的事项}
```

---

## 四、质量检查清单

方案完成后自查：
- [ ] 每个任务有明确的 owner 和依赖
- [ ] 每个任务可独立验收
- [ ] 数据模型覆盖了所有涉及的实体
- [ ] API 草案符合 api-standard.md
- [ ] 考虑了边界条件（空数据、错误、权限）
- [ ] 所有不确定的地方标注了 `[待确认]`
- [ ] 验收条件是具体可测试的，不是"功能正常运行"
