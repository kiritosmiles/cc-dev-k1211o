# Code Review Skill

代码审查方法论。由 reviewer agent 调用，提供系统化的审查步骤。

---

## 一、审查流程

```
读取 task 文件 + changed_files
    │
    ▼
① 架构审查 — 变更是否破坏模块边界
    │
    ▼
② 安全审查 — 逐文件检查安全风险
    │
    ▼
③ 性能审查 — 识别 N+1、重复请求、内存泄漏
    │
    ▼
④ 代码质量审查 — 可读性、复杂度、重复
    │
    ▼
⑤ 规范审查 — 命名、错误处理、日志、类型
    │
    ▼
⑥ 测试审查 — 测试覆盖是否充分
    │
    ▼
输出审查报告
```

---

## 二、各阶段检查清单

### 1. 架构审查

- [ ] 是否跨模块修改（如 frontend 改了 backend 文件）
- [ ] Controller 是否直接访问了数据库
- [ ] Service 是否存在强耦合
- [ ] 新增模块是否符合 architecture.md 定义的边界
- [ ] 依赖方向是否正确（高层不依赖低层实现细节）

### 2. 安全审查

- [ ] 是否存在 SQL 拼接（必须用参数化查询）
- [ ] 用户输入是否做了校验和转义（XSS 防护）
- [ ] 响应中是否暴露了内部错误信息
- [ ] 是否存在硬编码的密钥/token
- [ ] 敏感接口是否有权限检查
- [ ] 日志中是否记录了密码/token

### 3. 性能审查

- [ ] 循环中是否存在数据库查询（N+1）
- [ ] 是否存在重复的 API 请求（可合并/缓存）
- [ ] 大列表是否做了分页
- [ ] 是否存在不必要的数据加载（只取需要的字段）
- [ ] 前端是否存在不必要 re-render

### 4. 代码质量审查

- [ ] 函数是否超过 80 行
- [ ] 嵌套是否超过 3 层
- [ ] 是否存在魔法值（应提取为常量）
- [ ] 是否存在重复代码（超过 3 行相似逻辑应抽取）
- [ ] 类是否过大（超过 300 行）
- [ ] 命名是否清晰（变量/函数名应表达意图）
- [ ] 是否存在隐式依赖/隐式状态

### 5. 规范审查

- [ ] 命名是否符合规范（camelCase/PascalCase/UPPER_CASE）
- [ ] 是否有错误处理（不吞异常）
- [ ] 错误格式是否符合 api-standard.md
- [ ] 是否记录了关键日志
- [ ] 是否有类型标注

### 6. 测试审查

- [ ] 是否覆盖了正常流程
- [ ] 是否覆盖了异常流程
- [ ] 是否覆盖了边界条件
- [ ] 是否有权限相关测试

## 三、严重级别定义

| 级别 | 定义 | 处理方式 |
|------|------|---------|
| **Critical** | 会导致系统崩溃、数据丢失、安全漏洞 | 必须修复才能合入 |
| **Major** | 会影响功能正确性、明显违反规范 | 应该修复 |
| **Minor** | 可读性、命名、小优化建议 | 建议修复，不阻塞 |

---

## 四、常见问题速查

| 问题 | 级别 | 示例 |
|------|------|------|
| SQL 拼接 | Critical | `query = "SELECT * FROM users WHERE id=" + user_id` |
| 暴露内部错误 | Major | `response: { error: err.stack }` |
| 硬编码密钥 | Critical | `const API_KEY = "sk-xxx"` |
| 无分页 | Major | `SELECT * FROM orders`（100万行） |
| N+1 查询 | Major | 循环中逐条查关联数据 |
| 缺少错误处理 | Major | `try { ... } catch(e) {}` |
| 函数过长 | Minor | 超过 80 行的函数 |
| 魔法值 | Minor | `if (status === 3)` 而非 `if (status === STATUS_PAID)` |
